# 讨论-三省六部审核机制

> 设计专家小皮 | 2026-03-19

---

## 一、审核机制设计建议

### 1. 核心角色定义

| 角色 | 职责 | 适用场景 |
|------|------|----------|
| **中书省（起草）** | 方案/文档的发起者，负责起草初稿 | 产品需求、设计方案、代码PR |
| **门下省（审核）** | 审核把关，质疑/优化/否决 | 所有待发布内容 |
| **尚书省（执行）** | 执行最终批准的方案 | 研发实现、设计产出、部署上线 |

### 2. 审核流程

```
中书省起草 → 门下省审核 → 尚书省执行 → 反馈闭环
```

### 3. 设计小皮的审核点

作为设计专家，我在"门下省审核"环节关注：

- **用户体验**：方案是否符合用户习惯？
- **视觉一致性**：是否符合设计规范？
- **交互合理性**：操作流程是否顺畅？
- **边界情况**：异常状态是否处理？

---

## 二、我的角色定位

### 门下省审核官（UX视角）

**审核关注点：**
1. 用户体验是否友好
2. 视觉设计是否一致
3. 交互逻辑是否合理
4. 信息架构是否清晰

**否决权触发条件：**
- 严重影响用户体验
- 违背设计规范
- 交互逻辑有重大缺陷

## 三、设计建议

### 1. 分级审核机制

| 等级 | 说明 | 审核深度 |
|------|------|----------|
| L1 | 例行事项 | 快速确认 |
| L2 | 重要事项 | 详细审核 |
| L3 | 重大事项 | 深度讨论+多方会审 |

### 2. 审核反馈模板

```markdown
## 门下省审核意见

**方案**：xxx
**审核人**：设计小皮
**时间**：2026-03-19

### ✅ 通过
- 优点：xxx

### ⚠️ 建议优化
- 问题：xxx
- 建议：xxx

### ❌ 否决
- 原因：xxx
- 影响：xxx
```

### 3. 快速反馈机制

- L1 审核：1小时内反馈
- L2 审核：4小时内反馈
- L3 审核：24小时内组织讨论

---

## 四、适合设计的审核场景

1. **产品需求发布前** → 审核用户体验合理性
2. **设计方案交付前** → 审核视觉和交互规范性
3. **新功能上线前** → 审核整体用户体验
4. **重大改版前** → 深度参与多方会审
