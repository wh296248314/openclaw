## 一、任务看板四列状态流转审查

### 1.1 现状：四列设计

| 列名 | 原型状态 | 流转动作 |
|------|----------|----------|
| 📥 待派发 | 静态展示 | 派发 |
| ⚡ 进行中 | 静态展示 | 完成 / 阻塞 |
| 🔍 待验收 | 静态展示 | 通过 / 不通过 |
| ✅ 已完成 | 静态展示 | 无（终态） |

### 1.2 ✅ 已覆盖的流转路径

```
待派发 → [派发] → 进行中
进行中 → [完成] → 待验收
进行中 → [阻塞] → (状态未知)
待验收 → [通过] → 已完成
待验收 → [不通过] → (未定义)
```

### 1.3 🔴 缺失的场景

#### 问题1：阻塞任务无法恢复
- **现象**：`blockTask()` 只改变了按钮文字和样式，没有实际状态变更
- **影响**：任务被阻塞后无法解除，回到可执行状态
- **建议**：增加"解除阻塞"按钮，流转到"进行中"

#### 问题2：不通过后状态未定义
- **现象**："不通过"后任务去向不明
- **当前行为**：原型只显示 toast "打回修改"，无实际流转
- **建议**：明确"不通过"→返回"进行中"重新修改，或→返回"待派发"

#### 问题3：任务取消流程缺失
- **现象**：无取消按钮，无取消后的状态处理
- **建议**：增加"取消"列或"已取消"终态

#### 问题4：派发失败无重试机制
- **原型显示**：`派发状态: gateway-offline`，但无重试按钮
- **建议**：增加"重试派发"按钮

#### 问题5：缺少状态变更时间戳
- **现象**：任务卡片无创建时间、更新时间、状态变更记录
- **建议**：增加时间字段，支撑任务追踪

### 1.4 测试用例设计建议

```javascript
// TC-001: 正常流转
待派发 → 派发 → 进行中 → 完成 → 待验收 → 通过 → 已完成

// TC-002: 阻塞恢复
进行中 → 阻塞 → (无恢复路径) ❌ 需修复

// TC-003: 验收不通过
待验收 → 不通过 → 进行中 ❌ 未定义

// TC-004: 派发失败
待派发 → 派发失败 → 显示错误 → 重试派发 ❌ 无重试UI
```

## 二、旨意审批流程审查

### 2.1 标准流程（PRD定义）

```
皇上(皮休) → 中书省(起草) → 门下省(审议) → 尚书省(派发) → 六部(执行) → 尚书省(汇总)
```

### 2.2 ✅ 已覆盖场景

- 旨意创建和列表展示 ✅
- 各省阶段流转（✓图标展示）✅
- 当前阶段高亮显示 ✅
- 操作按钮：叫停/取消/推进 ✅

### 2.3 🔴 缺失场景

#### 问题6：审批驳回/打回功能缺失
- **现象**：原型只有"推进到下一步"，无"驳回"或"打回"按钮
- **实际业务**：门下省审核不通过 → 应打回中书省重新起草
- **建议**：增加"驳回"按钮，流转到上一个省部

#### 问题7：旨意状态不完整
- **缺失状态**：
  - 驳回状态（被打回的旨意）
  - 暂停状态（被叫停后）
  - 终止状态（被取消后）
- **原型只展示**：活跃 / 归档 / 全部，无终止态

#### 问题8：六部并行执行未体现
- **PRD描述**：六部可并行执行（兵部研发、礼部设计、刑部测试并行）
- **原型现状**：六部显示为单一节点，无并行展示
- **建议**：六部阶段应展示多个子任务卡片

#### 问题9：旨意归档条件不明确
- **现象**：原型有"归档"筛选，但归档规则未定义
- **问题**：任务完成多久后归档？手动还是自动？

### 2.4 审批流测试用例

```javascript
// TC-010: 正常审批通过
皇上 → 中书省(起草) → 门下省(通过) → 尚书省(派发) → 六部(执行) → 尚书省(汇总) → 归档

// TC-011: 门下省驳回
皇上 → 中书省(起草) → 门下省(驳回) → ? → 应打回中书省 ❌ 未定义

// TC-012: 皇上叫停
皇上 → (叫停) → 状态变为"已叫停" → 无后续流转 ❌ 缺少终止态
```

## 三、验收标准审查

### 3.1 现状

PRD中**完全没有验收标准章节**。仅在"版本历史"和"后续迭代"中有功能描述，无可量化指标。

### 3.2 🔴 需要补充的验收标准

| 功能模块 | 当前状态 | 缺失内容 |
|----------|----------|----------|
| 旨意创建 | 描述性 | 创建成功的判断条件是什么？旨意号格式？ |
| 任务派发 | 描述性 | 派发成功的判断条件？派发失败重试几次？ |
| 任务完成 | 描述性 | 谁来确认完成？完成的标准是什么？ |
| 验收流程 | 描述性 | 验收不通过的条件？打回次数限制？ |
| 归档规则 | 无 | 自动归档还是手动？保留多久？ |
| 阻塞检测 | 无 | 任务多久无响应算阻塞？超时阈值？ |
| 叫停恢复 | 无 | 被叫停的任务能恢复吗？恢复流程？ |

### 3.3 建议补充格式

```markdown
### 2.3 任务派发验收标准

**前置条件**：任务状态为"待派发"，目标部门在线

**验收条件**：
1. 派发后任务状态变为"进行中"
2. 任务卡片显示派发时间
3. 派发失败时显示错误信息，且任务保留在"待派发"
4. 派发失败3次后自动标记"派发异常"，发送通知

**验收方式**：
- 自动化：检查任务状态字段变更
- 手动：操作日志显示派发记录
```

## 六、综合测试风险评估

### 6.1 高风险项（阻塞发布）

1. **验收标准缺失** - 无法制定测试计划，无通过标准
2. **状态流转不完整** - 阻塞/不通过/取消流程缺失，核心功能闭环未形成
3. **审批驳回缺失** - 三省决策机制不完整

### 6.2 中风险项（影响质量）

4. **后端API未实现** - 无法进行端到端测试
5. **数据持久化未验证** - 刷新页面数据丢失
6. **归档规则不明确** - 数据清理策略缺失

### 6.3 低风险项（可迭代）

7. **UI细节优化** - 空状态、动画、表单验证
8. **六部并行展示** - 优化展示效果
9. **实时通知** - 可降级为轮询
