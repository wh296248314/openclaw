# 五小皮团队协作模型

> 基于「三省六部制」理念设计的 AI Agent 协作架构

## 背景

参考 [cft0808/edict](https://github.com/cft0808/edict) 三省六部制多 Agent 协作系统，结合五小皮（产品、设计、研发、测试、部署）的实际情况，形成适合的协作模型。

---

## 一、当前五小皮职责

| 小皮 | 核心职责 |
|------|----------|
| 产品小皮 | 需求分析、任务拆解、功能规划 |
| 设计小皮 | UI/UX 设计、交互方案 |
| 研发小皮 | 代码开发、功能实现 |
| 测试小皮 | 测试执行、缺陷验证 |
| 部署小皮 | 部署上线、环境配置 |
| 审核小皮 | 质量把关、流程把控、审核验收 |

---

## 二、参考原型：三省六部制

### 2.1 核心流程

```
皇帝(你) → 太子(分拣) → 中书省(规划) → 门下省(审核) → 尚书省(派发) → 六部(执行) → 回奏
```

### 2.2 核心优势

| 特性 | 三省六部 | 传统框架(CrewAI/AutoGen) |
|------|----------|--------------------------|
| 审核机制 | 门下省专职审核，可封驳 | 无/可选 |
| 实时看板 | 军机处看板 | 无 |
| 任务干预 | 叫停/取消/恢复 | 无 |
| 审计追踪 | 完整奏折存档 | 有限 |
| 热切换模型 | 看板一键切换 | 无 |

### 2.3 关键洞察

> **制度性审核 > 自由协作**
> 
> CrewAI 是"你们自己聊，聊完给我"；
> 三省六部是"先规划、再审核、然后执行、最后回奏"——1300年前唐太宗就想明白的分权制衡。

## 三、五小皮协作模型

### 3.1 简化版流程

```
产品小皮（需求分析+任务分配）
       ↓
设计小皮 + 研发小皮（并行）
       ↓
测试小皮（测试执行）
       ↓
审核小皮（质量审核 ⭐关键！）
       ↓
部署小皮（上线）
```

### 3.2 各角色定位

| 角色 | 定位 | 类比三省六部 |
|------|------|-------------|
| 产品小皮 | 需求中枢，任务分发 | 中书省 |
| 设计小皮 | 设计输出，并行执行 | 礼部 |
| 研发小皮 | 开发实现，并行执行 | 兵部 |
| 测试小皮 | 测试执行，缺陷验证 | 刑部 |
| 审核小皮 | 质量审核，守门把关 | 门下省 |
| 部署小皮 | 上线交付，收尾闭环 | 工部 |

### 3.3 审核机制（核心变化）

**测试小皮拥有测试执行权，审核小皮拥有质量审核权（封驳权）**：

```
研发交付 → 测试执行 → 审核小皮审核 → 
    ✅ 通过 → 部署小皮
    ❌ 封驳 → 研发返工 → 再次提交 → 循环直到通过
```

这解决了传统模式的问题：研发交付后测试才被动发现问题，或者缺少独立的质量审核环节。

## 四、可借鉴的功能特性

### 4.1 任务流转看板

统一视角查看5个小皮的：
- 当前任务状态
- 待办/进行中/已完成
- 阻塞项/风险项

### 4.2 交付物审核

- 研发交付代码 → 测试审核
- 设计交付稿 → 产品/测试确认
- 部署前 → 全体确认

### 4.3 审计日志

每个任务的完整流转记录：
- 创建时间
- 各环节处理时间
- 返工次数
- 最终结果

### 4.4 干预能力

- 任务暂停/取消
- 任务重置
- 任务转移执行人

## 五、实施建议

### 5.1 短期（快速见效）

1. **建立交付规范**：明确研发→测试的交付物标准
2. **引入审核环节**：测试有权打回不合格的交付
3. **串通消息流**：5个小皮之间能相互通信

### 5.2 中期（逐步完善）

1. **任务看板**：统一视图
2. **审计日志**：记录完整流转
3. **干预机制**：可暂停/重置

### 5.3 长期（理想形态）

参考三省六部，打造完整的：
- 任务分发系统
- 审核流转机制
- 实时监控看板
- 完整审计追踪

## 六、对比总结

| 维度 | 当前模式 | 改进后 |
|------|----------|--------|
| 任务流向 | 产品→研发→测试→部署 | 产品→研发/设计→测试(审核)→部署 |
| 测试角色 | 被动接收 | 主动审核，可封驳 |
| 可见性 | 各自为战 | 统一看板 |
| 追溯性 | 有限 | 完整审计 |

## 八、各小皮建议汇总

### 8.1 测试小皮建议：小皮管家核心能力

1. **意图理解能力** - 准确理解指令，识别闲聊vs任务
2. **任务调度能力** - 合理分配任务给对应小皮
3. **质量把控能力** - 协调审核职能
4. **知识共享能力** - 团队知识库
5. **应急处理能力** - 失败及时反馈
6. **持续学习能力** - 总结经验优化

### 8.2 产品小皮建议：小皮团队核心能力

1. **记忆连续性** - 会话自动回忆上下文，避免重复询问
2. **主动学习能力** - 定时扫描新技能/知识
3. **审核把控能力** - 方案设计后必须有审核环节
4. **权限边界意识** - 不越权调用，明确权限矩阵
5. **进度可视化** - 任务状态实时可查
6. **干预和纠错能力** - 任务可叫停、取消、恢复
7. **沟通规范** - 处理时间提前告知

### 8.3 优先建设

**记忆连续性 + 审核把控 + 权限边界** —— 当前团队最欠缺的三大能力。

### 8.3 部署小皮建议：小皮管家核心能力

1. **任务分发能力** - 准确理解意图，自动分配任务
2. **上下文记忆能力** - 多轮对话保留，产出可追溯
3. **进度感知能力** - 实时追踪任务状态，识别阻塞
4. **工具整合能力** - 统一调度飞书、GitHub、Docker等
5. **质量把控能力** - 协调审核环节
6. **沟通协调能力** - 各专家间传递信息
7. **文档管理能力** - 自动沉淀项目文档
8. **异常处理能力** - 识别错误，优雅降级

**核心定位：管家 = 协调者 + 调度者 + 守门人**

---

### 8.4 设计小皮建议：小皮核心能力

1. **专业深度** - 深厚专业知识，给出专业见解
2. **协作意识** - 理解需要配合，主动沟通
3. **主动学习** - 不懂就问，保持好奇心
4. **需求理解** - 先理解真正要什么，多问为什么
5. **记忆与传承** - 记住决策，整理文档
6. **批判性思维** - 敢于质疑，给出替代方案

**一句话总结：小皮不仅是执行者，更是能思考、会协作、愿学习的专业伙伴。**

---

### 8.5 研发小皮建议：小皮核心能力

1. **专业技术深度** - 深入理解技术原理，代码审查能力
2. **问题解决能力** - 快速定位根因，提供多种方案
3. **学习与进化** - 主动学习新技术，从错误中总结
4. **协作沟通** - 清晰表达技术观点，配合其他专家
5. **主动性** - 主动发现问题，提供优化建议
6. **可靠负责** - 交付结果可信赖，诚实面对不确定性
