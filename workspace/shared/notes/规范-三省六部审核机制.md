# 三省六部审核机制 - 正式方案

**版本**: v1.0
**日期**: 2026-03-19
**制定人**: 小皮管家（汇总各小皮讨论意见）
**审核人**: 皮休/晗哥

---

## 一、核心定位

将"三省六部"古代官制融入小皮团队协作，形成"规划→审核→执行→监督"的闭环流程。

### 核心价值
- **质量守门**：审核前置，减少返工
- **职责清晰**：各司其职，避免越位
- **效率平衡**：分级管理，不一刀切
- **全程留痕**：记录可追溯

---

## 二、角色定义（三省）

| 角色 | 小皮 | 类比 | 核心职责 |
|------|------|------|----------|
| **中书省** | 产品小皮 | 起草诏令 | 需求分析、任务规划、分发协调 |
| **门下省** | 审核小皮 | 审核封驳 | 质量把关、风险识别、最终审核 |
| **尚书省** | 小皮管家 | 执行统筹 | 协调派发、进度跟踪、资源调配 |

### 六部执行

| 角色 | 小皮 | 职责 |
|------|------|------|
| 吏部 | 研发小皮 | 技术方案、代码质量、安全审核 |
| 礼部 | 设计小皮 | 用户体验、视觉规范、交互审核 |
| 刑部 | 测试小皮 | 功能验证、边界测试、质量评估 |
| 工部 | 部署小皮 | 部署实施、运维保障、监控优化 |

### 最终决策

> **皇帝 = 皮休/晗哥** — 拥有最终决策权和否决权

---

## 三、分级审核机制

### 审核等级

| 等级 | 触发条件 | 审核流程 | 时效要求 | 审核人 |
|------|----------|----------|----------|--------|
| **L1** | 例行事项、格式检查 | 自检即可 | 即时 | 执行者本人 |
| **L2** | 跨域交付、方案讨论 | 执行者 → 对应专家 | ≤4小时 | 对应专家小皮 |
| **L3** | 新功能上线、重大改版 | 中书省 → 门下省 | ≤24小时 | 审核小皮 |
| **L4** | 战略决策、重大变更 | 全流程 → 皇帝 | 皮休决策 | 皮休 |

### L4 强制触发条件（必须皮休决策）
- 涉及预算、资源重大投入
- 跨团队协作事项
- 新技术选型或架构变更
- 用户数据相关决策

---

## 四、审核流程

### 标准流程（中大型任务）

```
1. 皮休下旨（提出需求/任务）
       ↓
2. 中书省起草（产品小皮分析需求、拆解任务）
       ↓
3. 标注审核等级（L1-L4）
       ↓
4. L2+ 进入审核流程
   - L2：对应专家审核
   - L3：审核小皮审核
   - L4：提交皮休决策
       ↓
5. 通过后，尚书省派发执行（六部执行）
       ↓
6. 执行完成，回奏皇上（汇报结果）
```

### 简化流程（小任务）

```
皮休下旨 → 产品拆分 → L1自检 → 执行 → 回奏
```

## 七、时效规则

| 任务类型 | 审核时效 | 超时处理 |
|----------|----------|----------|
| L1 | 即时 | - |
| L2 | ≤4小时 | 催办，逾期自动通过 |
| L3 | ≤24小时 | 自动升级皮休 |
| L4 | 皮休安排 | - |

> ⚠️ 紧急任务可申请加急，审核人需在2小时内响应

## 八、申诉机制

### 申诉条件
- 对审核结果有异议
- 认为审核标准不合理
- 存在理解分歧

### 申诉流程
```
被审核方提出申诉 → 小皮管家协调 → 
- L2/L3申诉 → 提交皮休裁决
- 皮休拥有一票决定权
```

## 九、与edict系统结合

### edict 定位
- **任务看板**：展示任务流转状态
- **审核记录**：留存审核意见
- **可视化**：让皮休一目了然

### 建议结合点
1. 重要决策先提交 edict 系统
2. 审核小皮通过系统审核
3. 皮休在系统里做最终决策
4. 全程留痕可追溯
