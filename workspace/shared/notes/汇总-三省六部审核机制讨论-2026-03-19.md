## 二、机制设计共识

### 角色定位（三省）

| 角色 | 小皮 | 职责 |
|------|------|------|
| **中书省** | 各执行小皮 | 方案起草、执行 |
| **门下省** | 审核小皮 | 质量审核、封驳 |
| **尚书省** | 小皮管家 | 任务派发、协调 |
| **皇帝** | 皮休/晗哥 | 最终决策 |

### 审核流程

```
皮休下旨 → 中书省起草 → 门下省审核 → 尚书省执行 → 回奏皇上
```

### 审核触发条件（共识）

| 小皮 | 建议触发条件 |
|------|-------------|
| 部署 | 预估时间>30分钟的任务 |
| 审核 | 重大决策、跨团队协作、新功能上线 |
| 研发 | 新项目立项、重大技术变更 |
| 设计 | 需求发布前、设计交付前 |

### 审核时效（共识）

| 等级 | 反馈时间 | 适用场景 |
|------|----------|----------|
| L1 快速 | 1小时内 | 例行事项 |
| L2 详细 | 4小时内 | 重要事项 |
| L3 深度 | 24小时内 | 重大事项 |

## 四、共同建议

### 1. 审核前置
- 问题发现越早，修复成本越低
- 建议方案设计阶段就开始审核

### 2. 审核标准明确化
- 建立各小皮的产出规范checklist
- 避免审核流于形式

### 3. 与edict系统结合
- 重要决策提交edict系统
- 审核记录全程留痕可追溯

### 4. 效率与质量平衡
- 不是所有任务都需要完整审核流程
- 分级管理，小任务快速通过

### 5. 申诉机制
- 对审核结果有异议可向皮休申诉
- 皮休拥有一票决定权
