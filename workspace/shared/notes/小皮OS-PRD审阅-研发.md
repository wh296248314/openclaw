## 一、技术可行性评估

### 1. HTML5单文件应用 ✅ 基本可行（需改进）

**原型分析**：
- 文件大小：2724行，约80KB
- 技术栈：纯HTML + CSS Variables + 原生JavaScript
- 无任何外部依赖，无构建工具

**优点**：
- ✅ 单文件分发简单，双击即可运行
- ✅ CSS Variables统一管理主题色，维护成本低
- ✅ 原型阶段快速迭代，无需构建

**问题**：
- ⚠️ **样式与逻辑混杂**：2724行混在一起，产品/设计/研发无法协作
- ⚠️ **无法模块化**：2000行CSS + 2000行JS在单文件中，难以维护
- ⚠️ **无代码规范**：无ESLint/Prettier，团队协作困难

**建议**：
- 开发阶段使用Vue/React + 构建工具（Vite）
- 正式发布时可打包为单文件或使用CDN分发
- PRD中应明确标注"原型为单文件，开发版本用Vue3"

## 二、实时动态每30秒更新 ⚠️ 方案可行但实现错误

**PRD描述**：实时动态滚动更新，每30秒自动刷新

**原型实际实现**：
```javascript
// 原型中的实现（错误的模拟方式）
setInterval(addLiveEvent, 30000);  // ✅ 定时器正确

function addLiveEvent() {
  // 轮播预定义的mock数组，不是真实API
  const event = liveEvents[liveIndex % liveEvents.length];
  // ...
}
```

**问题**：
- ❌ **数据是Mock的**：原型中只有5条预定义事件，循环播放
- ❌ **不是真实API调用**：没有fetch/axios请求
- ❌ **不是真正的"实时"**：只是页面刷新模拟
- ⚠️ **没有连接edict后端**：原型完全独立，无数据源

**正确实现方案**：

```
方案A：轮询（适合低频更新）
  setInterval(() => fetch('/api/live-status'), 30000)

方案B：SSE（Server-Sent Events，适合实时性）
  const es = new EventSource('/api/live-stream')

方案C：WebSocket（适合高频交互）
  const ws = new WebSocket('ws://host/api/ws')

推荐方案：方案A + 方案B结合
- 日常用SSE保持长连接
- 断线重连用轮询兜底
```

**PRD缺失**：
- 未定义数据源（edict的哪个API？）
- 未定义数据格式
- 未定义更新失败的处理策略

## 三、数据存储方案 ❌ 严重缺失

**PRD描述**：无明确数据存储方案

**原型实际实现**：
- ❌ **完全没有数据持久化**
- ❌ **没有LocalStorage**
- ❌ **没有后端API调用**
- ❌ **所有数据都是硬编码**

**现状对比**：

| 功能 | PRD描述 | 原型实现 | 差距 |
|------|---------|----------|------|

| 任务数据 | 四列看板 | 静态卡片 | 需API接入 |

| 档案馆 | 上传文档 | 只弹Toast | 需API接入 |
| 配置中心 | 操作日志 | 空 | 需API接入 |

**建议数据架构**：

```
┌─────────────────────────────────────────────┐
│              小皮OS 前端                     │
│  (Vue3 / 旨意看板 / 任务看板 / 官员总览)      │
└─────────────────┬───────────────────────────┘
                  │ fetch / SSE / WebSocket
                  ▼
┌─────────────────────────────────────────────┐
│           edict API 网关                     │
│  /api/live-status  → 实时状态                │
│  /api/tasks        → 任务列表                │
│  /api/officials    → 官员数据                │
│  /api/create-task  → 创建任务                │
│  /api/advance-state → 推进状态               │
└─────────────────┬───────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────┐
│        OpenClaw Runtime Sessions             │
│     (各小皮的实时会话数据)                    │
└─────────────────────────────────────────────┘
```

**LocalStorage用途**（仅用于前端）：
- 用户偏好设置（主题、语言）
- 草稿缓存（未提交的旨意/任务）

## 五、技术实现的难点和风险

### 🔴 高风险

**1. 后端API依赖（edict）**

当前edict存在多个API 404错误：
- `/api/tasks` → 404 ❌
- `/api/create-task` → 404 ❌
- `/api/advance-state` → 404 ❌

**影响**：小皮OS的核心功能（下旨、流转、归档）完全依赖这些API，如果edict不修复，小皮OS无法运行。

**建议**：
- PRD应明确要求"edict v1.1需提供完整的REST API"
- 小皮OS开发优先级 = edict API修复优先级

**2. 数据源统一**

小皮OS和edict现在各有一份数据，两边不同步。

**建议**：
- 明确edict为唯一数据源
- 小皮OS只读不写（展示用）
- 所有操作通过edict API

**3. 实时性与性能**

7个Tab + 20个任务 + 7个官员，每30秒轮询所有数据会产生性能问题。

**建议**：
- 使用SSE长连接代替轮询
- 差异化刷新（仪表盘30s，官员总览60s）

### 🟡 中风险

**4. 单文件原型的局限性**

原型是纯展示用的，无法体现：
- 表单验证逻辑
- 错误处理
- 断线重连
- 多用户并发

**建议**：开发版必须用完整框架，原型仅用于UI验证。

**5. v1.1/v1.2功能依赖后端开发**

朝堂议政、奏折阁、天下要闻等v1.1/v1.2功能需要：
- court-discuss API（已存在，需完善）
- 外部数据源接入（RSS/API）
- 复杂的状态管理

**建议**：小皮OS v1.0聚焦展示，v1.1再扩展。
