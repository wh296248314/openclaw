## 二、技能管理体系

### 1. 技能分类（皮休要求）

| 类型 | 说明 | 例子 | 使用时机 |
|------|------|------|----------|
| **快速完成型** | 内置/已配置技能，直接调用 | feishu-doc、weather、file-organizer | 问题明确、有现成技能时 |
| **自行创建型** | 根据需求自己创建的技能 | kubectl速查、Semgrep | 频率高、通用性强时 |
| **组合技能型** | 多个技能组合使用 | 自动备份+通知、文档同步+飞书通知 | 复杂任务需要多步骤时 |

### 2. 问题-技能映射表（皮休关心的问题类型）

| 皮休问题类型 | 对应技能 | 技能类型 | 使用频率 |
|--------------|----------|----------|----------|
| **飞书文档操作** | feishu-doc | 快速完成型 | 高 |
| **飞书权限管理** | feishu-perm | 快速完成型 | 中 |
| **飞书知识库** | feishu-wiki | 快速完成型 | 中 |
| **天气查询** | weather | 快速完成型 | 中 |
| **文件整理** | file-organizer | 快速完成型 | 高 |
| **代码安全审计** | Semgrep（待创建） | 自行创建型 | 中 |
| **K8s运维** | kubectl速查 | 自行创建型 | 中 |
| **OpenClaw节点连接** | node-connect | 快速完成型 | 低 |
| **技能管理** | clawhub | 快速完成型 | 中 |
| **健康检查** | healthcheck | 自行创建型 | 中 |
| **网页内容提取** | summarize | 快速完成型 | 高 |
| **网页搜索** | web_search | 快速完成型 | 高 |

### 3. 技能决策树

```
遇到问题
    ↓
是否是常见问题类型？
    ↓
┌────────────────────────────────────────────┐
│ 是                                         │
│ ↓                                         │
│ 查 skills-index.md 映射表                   │
│ ↓                                         │
│ 有现成技能？                                │
│   ├─ 是 → 直接调用 → 执行 → 记录使用         │
│   └─ 否 → 进入下方流程                     │
└────────────────────────────────────────────┘
    ↓
┌────────────────────────────────────────────┐
│ 否                                         │
│ ↓                                         │
│ 分析问题：                                 │
│ · 需要什么能力？                           │
│ · 频率高不高？                             │
│ · 是否可组合现有技能？                      │
│ ↓                                         │
│ 决策：                                     │
│ · 频率高+通用 → 创建自建技能                │

│ · 偶尔一次 → 手动解决，记录为"待建技能"     │
└────────────────────────────────────────────┘
```

### 4. 技能价值评估维度

| 维度 | 评估问题 | 权重 |
|------|----------|------|
| **使用频率** | 这个问题多久出现一次？ | 高 |
| **解决效率** | 用技能能节省多少时间？ | 高 |
| **通用性** | 这个问题其他小皮也会遇到吗？ | 中 |
| **可复用性** | 这个技能还能解决什么问题？ | 中 |
| **学习成本** | 创建这个技能需要多少工作量？ | 低 |

### 5. 技能创建流程

```
发现需求 → 评估价值
     ↓
  价值高？
     ↓
┌────────────────────────────────────────────┐
│ 是                                         │
│ ↓                                         │
│ 创建技能：                                 │
│ 1. 编写 SKILL.md                          │
│ 2. 编写脚本/配置                           │
│ 3. 测试验证                               │
│ 4. 记录到 skills-index.md                  │
│ 5. 同步给各小皮                            │
└────────────────────────────────────────────┘
    ↓
  通知皮休 → 纳入正式技能库
```

## 三、每日技能学习（节点0）

### 执行时间
- 每天 9:30 前完成检查

### 检查流程
1. 检查 `shared/notes/` 是否有当日技能分享
2. 如未提交，@all 催促
3. 9:45 仍未提交，向皮休汇报

### 学习要求
| 项目 | 要求 |
|------|------|
| 学习时间 | 每天9:30前完成 |
| 提交路径 | `shared/notes/技能分享-{小皮}-{YYYY-MM-DD}.md` |
| 内容要求 | 技能名称、特点(3点)、应用场景、学习资源 |
| 技能价值 | 分析是否可解决某类问题，是否纳入映射表 |

## 八、自动补充规范机制

### 触发条件
当遇到无现有规范可遵循的场景时

### 执行流程
1. 分析任务需求，判断需要什么规范
2. 搜索相关规范
3. 制定临时规范
4. 记录到当日 memory
5. 告知皮休确认
6. 确认后 → 加入正式规范库
