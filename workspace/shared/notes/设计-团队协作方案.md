# 团队协作模型 - 三省六部制

> 基于 cft0808/edict 三省六部模型设计的团队协作方案

---

## 背景

2026-03-16，皮休分享了 [cft0808/edict](https://github.com/cft0808/edict) 项目，我们研究后认为可以借鉴其"制度性审核 + 实时看板"的核心思想，优化小团队协作模式。

---

## 团队角色映射

| 小皮 | 三省六部职位 | 核心职责 |
|------|-------------|----------|
| 产品小皮 | 中书省 | 需求规划、任务拆解、把控方向 |
| 设计小皮 | 礼部 | 界面设计、交互设计、用户体验 |
| 研发小皮 | 工部 | 代码开发、功能实现 |
| 测试小皮 | 门下省 | 质量审核、把关、验收 |
| 部署小皮 | 兵部 | 部署运维、环境配置 |
| 皮休 | 皇上 | 最终决策、任务下发、验收 |

---

## 简化版协作流程

```
皮休(皇上) 
    ↓
产品小皮(中书省) → 需求规划、任务拆解
    ↓
┌─────────────────────────────────────┐
│  设计/研发/测试/部署 → 并行执行      │
│  • 设计小皮(礼部) → 设计稿输出       │
│  • 研发小皮(工部) → 代码开发         │
│  • 部署小皮(兵部) → 部署配置         │
└─────────────────────────────────────┘
    ↓
测试小皮(门下省) → 质量审核、不合格打回
    ↓
皮休 → 最终验收
```

---

## 核心机制

### 1. 门下省审核机制（测试小皮）

- **设计审核**：测试小皮参与设计评审，验证设计稿可行性
- **代码审核**：测试小皮审核代码质量和测试覆盖率
- **部署审核**：测试小皮验证部署包和配置
- **打回权**：不合格可以打回重做，直到达标

### 2. 任务看板

任务状态流转：
- 待规划 → 已规划 → 执行中 → 审核中 → 已完成

| 状态 | 说明 |
|------|------|
| 待规划 | 新需求，等待产品小皮规划 |
| 已规划 | 需求已拆解，待设计/开发 |
| 设计中 | 设计小皮正在输出设计稿 |
| 开发中 | 研发小皮正在编码 |
| 部署中 | 部署小皮配置环境 |
| 审核中 | 测试小皮质量审核 |
| 已完成 | 通过审核，交付完成 |

### 3. 审计追踪

- 每个任务有完整的流转记录
- 设计决策、代码变更、测试结果都可追溯
- 方便复盘和责任认定

## 有价值的功能借鉴

### ⭐⭐⭐ 门下省审核
测试小皮的强制质量把关，减少返工，提升交付质量

### ⭐⭐ 实时看板
任务状态清晰可见，清楚每个需求在哪一个环节

### ⭐ 审计追踪
设计决策可追溯，方便复盘和问题定位
