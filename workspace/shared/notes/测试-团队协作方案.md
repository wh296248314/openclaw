# 五小皮团队协作模型 - 基于三省六部制架构

> 文档创建时间：2026-03-16
> 作者：测试专家小皮
> 参考原型：cft0808/edict 三省六部制

---

## 背景

研究 [cft0808/edict](https://github.com/cft0808/edict) 项目后，发现其"三省六部制"多智能体协作架构非常适合用于五小皮团队的协作建模。

原模型核心创新：用1300年前唐代帝国制度实现AI多Agent分权制衡。

---

## 一、团队角色映射

### 1.1 核心映射方案

| 三省六部角色 | 职责 | 五小皮对应 | 核心能力 |
|-------------|------|-----------|----------|
| **太子** | 消息分拣、需求整理 | （主小皮/皮休） | 闲聊识别、旨意提炼 |
| **中书省** | 规划、拆解任务 | 产品小皮 | 需求分析、任务分解、方案设计 |
| **门下省** | 审核、封驳、质量把控 | 测试小皮 | 质量评审、风险识别、标准把控 |
| **尚书省** | 派发、协调、汇总 | 产品小皮（兼任） | 任务调度、进度跟踪、结果整合 |
| **礼部** | 文档、规范、设计 | 设计小皮 | UI/UX设计、技术文档、规范制定 |
| **兵部** | 代码、算法、工程 | 研发小皮 | 功能开发、Bug修复、代码审查 |
| **工部** | 部署、基建、运维 | 部署小皮 | Docker配置、CI/CD、自动化 |

### 1.2 精简版五部制流程

```
皮休(你) 
    ↓ 下旨/需求
产品小皮(中书省) → 规划任务、拆解子任务
         ↓
测试小皮(门下省) → 审核方案、可封驳 ← 【关键质量关卡】
         ↓ 通过
设计小皮(礼部)   → 设计规范
研发小皮(兵部)   → 代码实现 ← 并行
部署小皮(工部)   → 部署发布
         ↓
测试小皮(验收) → 最终质量把关
         ↓
产品小皮(尚书省) → 汇总回奏
```

## 二、各角色职责定义

### 2.1 产品小皮（中书省 + 尚书省）

**核心职责：**
- 需求分析与理解
- 任务拆解与规划
- 协调各小皮工作
- 汇总最终成果

**权限：**
- 可向所有角色发送任务
- 接收测试小皮的封驳意见
- 负责最终交付汇总

### 2.2 测试小皮（门下省）

**核心职责：**
- 审核方案质量
- 验证产出是否符合需求
- 可封驳不合格产出（打回重做）
- 最终验收把控

**权限：**
- 可审查所有产出
- 有权打回不合格方案/代码/设计
- 强制返工循环直到达标

**关键原则：**
> 门下省的核心理念：1300年前唐太宗想明白的道理——**不受制约的权力必然会出错**。

### 2.3 设计小皮（礼部）

**核心职责：**
- 界面设计规范
- 交互体验设计
- 技术文档撰写
- 设计一致性验证

**权限：**
- 接收产品小皮的设计任务
- 向测试小皮提交设计产出
- 与研发小皮协作设计实现

### 2.4 研发小皮（兵部）

**核心职责：**
- 功能代码实现
- Bug修复
- 代码审查
- 技术难题攻关

**权限：**
- 接收产品小皮的开发任务
- 与设计小皮协作实现
- 向测试小皮提交代码产出

### 2.5 部署小皮（工部）

**核心职责：**
- 部署流程自动化
- 基础设施配置
- CI/CD流水线维护
- 环境监控与告警

**权限：**
- 接收产品小皮的部署任务
- 与研发小皮协作部署
- 确保部署质量

## 三、任务流转规则

### 3.1 状态流转图

```
产品小皮(规划) 
    ↓ 提交审核
测试小皮(审核) → 封驳 ↓ 通过
    ↑          ↓
    ← 打回重做
         ↓
设计/研发/部署(并行执行)
         ↓
测试小皮(验收)
         ↓ 通过/打回
产品小皮(汇总) → 完成
```

### 3.2 状态说明

| 状态 | 说明 |
|------|------|
| 规划中 | 产品小皮正在分析需求、拆解任务 |
| 待审核 | 方案/产出等待测试小皮审核 |
| 审核中 | 测试小皮正在审查 |
| 封驳 | 测试小皮打回，需要重做 |
| 执行中 | 设计/研发/部署正在并行工作 |
| 验收中 | 测试小皮进行最终验收 |
| 已完成 | 任务完成，产品小皮汇总 |

### 3.3 状态转换规则

- 规划中 → 待审核：产品小皮提交方案
- 待审核 → 审核中：测试小皮开始审核
- 审核中 → 封驳：测试小皮打回
- 封驳 → 规划中：产品小皮重新规划
- 审核中 → 执行中：测试小皮通过
- 执行中 → 验收中：执行部门提交产出
- 验收中 → 封驳：验收不通过
- 验收中 → 已完成：验收通过

## 四、测试视角的价值分析

### 4.1 门下省模式的测试价值

**核心价值：**
1. **质量前置** - 在规划阶段就介入审核，避免后期返工
2. **强制关卡** - 不可绕过的质量检查点
3. **循环改进** - 被打回后重新流转，直到达标

### 4.2 测试挑战

1. **审核边界** - 测试小皮的"封驳"权力边界需要明确
2. **时序控制** - 并行执行时的依赖管理
3. **异常恢复** - 任务失败/打回后的流程恢复

### 4.3 测试机会

1. 多Agent协作流程测试
2. 状态机转换合法性测试
3. 任务干预操作测试（叫停/取消/恢复）
4. 并发执行时的数据一致性测试

## 五、待定问题

1. **尚书省角色** - 是否需要专人担任？还是产品小皮兼任？
2. **并行程度** - 设计与研发是否真的可以并行？还是需要一定顺序？
3. **封驳权限** - 测试小皮能封驳什么级别的质量问题？
4. **仲裁机制** - 如果对封驳有异议，如何解决？
