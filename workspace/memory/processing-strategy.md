# 分步处理策略

## 适用场景
1. **大容量信息**（>50KB 或 >1000行）
2. **复杂多步骤任务**
3. **预计处理时间 > 3分钟**
4. **超过模型上下文限制**
5. **需要外部工具配合的任务**

## 拆分原则

### 1. 信息拆分
```
原始大任务 → 分析结构 → 拆分子任务 → 分步执行 → 汇总结果
```

### 2. 容量控制
- **文本处理**：每步不超过 20KB 或 500行
- **文件处理**：大文件分块读取（offset/limit）
- **数据分析**：按时间/类别/区域分批处理

### 3. 时间控制
- **单步时长**：不超过 2分钟
- **总步骤数**：3-5步为宜，最多不超过10步
- **状态汇报**：每步完成后汇报进度

## 处理流程

### 第1步：任务评估
```markdown
🔍 **任务评估中...**
- 任务类型：[信息处理/文件操作/数据分析等]
- 预计规模：[大/中/小]
- 预计步骤：[3-5步]
- 预计总时长：[5-10分钟]
```

### 第2步：制定分步计划
```markdown
📋 **分步计划**
1. 步骤1：[具体内容] - 预计2分钟
2. 步骤2：[具体内容] - 预计2分钟  
3. 步骤3：[具体内容] - 预计2分钟
```

### 第3步：分步执行与汇报
```markdown
✅ **步骤1完成** (2/5)
- 完成了：[具体成果]
- 下一步：[步骤2内容]
- 预计完成时间：[时间]
```

### 第4步：最终汇总
```markdown
🎉 **任务完成汇总**
- 总步骤：[5步]
- 总耗时：[10分钟]
- 最终成果：[总结]
- 产出文件：[如有]
```

## 特殊情况处理

### 1. 遇到技术限制
```
⚠️ **遇到限制**
- 问题描述：[具体限制]
- 解决方案：[分步/换工具/简化]
- 建议步骤：[调整后的计划]
```

### 2. 处理失败
```
❌ **步骤失败**
- 失败原因：[具体原因]
- 已尝试：[尝试的方法]
- 备选方案：[替代方案]
- 是否需要人工介入？[是/否]
```

### 3. 超时处理
```
⏰ **处理超时**
- 当前进度：[已完成X%]
- 超时原因：[分析原因]
- 调整计划：[新的时间预估]
- 是否继续？[是/否]
```

## 示例：处理100KB文档

### 原始任务：分析100KB文档
**拆分方案：**
1. **步骤1**：读取文档前20KB，分析结构 (2分钟)
2. **步骤2**：读取20-40KB，提取关键信息 (2分钟)  
3. **步骤3**：读取40-60KB，继续分析 (2分钟)
4. **步骤4**：读取60-80KB，补充信息 (2分钟)
5. **步骤5**：读取80-100KB，汇总分析 (2分钟)
6. **步骤6**：生成最终报告 (2分钟)

**总耗时**：约12分钟，分6步完成

## 工具配合策略

### 1. 文件操作
- 大文件：使用 `read` 的 offset/limit 参数
- 多文件：分批处理，每批3-5个文件

### 2. 数据处理
- 数据库查询：分页查询，每页100条
- API调用：分批请求，设置合理间隔

### 3. 外部工具
- 复杂任务：使用 `sessions_spawn` 创建子代理
- 长时间任务：使用 `exec` 后台执行
